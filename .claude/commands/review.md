description: "変更されたファイルを 42 Norm v4.1 とプロジェクト規約に照らしてレビューする"
---

以下の手順でコードレビューを実施してください。

## 1. 変更差分の取得

```bash
git diff --name-only HEAD~1
```

で変更されたファイル一覧を取得し、各ファイルの内容を読んでください。

## 2. チェック項目

変更された各ファイルについて、以下を順にチェックしてください。
完全な Norm 仕様は docs/norm_v4.1.pdf を参照。

### 🔴 Norm 違反（致命的 — 評価 0 点）

#### フォーマット
- 関数が 25 行を超えていないか（波括弧 `{` `}` の行は除外してカウント）
- 1 ファイルに関数定義が 5 つを超えていないか（static 含む）
- 1 行が 80 文字を超えていないか（タブは 4 文字分としてカウント）
- インデントがスペースではなくタブ（4文字幅）になっているか
- 波括弧が単独の行にあるか（struct/enum/union 宣言部分は除く）
- 空行が連続していないか。空行に空白/タブが含まれていないか
- 行末に空白やタブがないか

#### 変数宣言
- 関数の途中で変数宣言していないか（冒頭のみ許可）
- 宣言と初期化を同一行で行っていないか（static/const は除く）
- 1 行に 2 つ以上の変数を宣言していないか
- 関数内の変数宣言が 5 つを超えていないか
- 変数宣言とコード本体の間に空行が 1 行あるか（他の空行は不可）
- ポインタの `*` が変数名に隣接しているか（`char *str` ○、`char * str` ✗）

#### 関数
- 引数が 4 つを超えていないか
- 引数なしの関数で `void` を省略していないか
- `return` の戻り値を丸括弧で囲んでいるか: `return (val);`
- プロトタイプの引数に名前があるか

#### 禁止構文
- `for` ループを使っていないか（`while` に置き換え）
- `do...while` を使っていないか
- `switch`/`case` を使っていないか
- `goto` を使っていないか
- 三項演算子 `?:` を使っていないか
- 可変長配列 (VLA) を使っていないか

#### 命名規則
- struct名が `s_` で始まるか、typedef名が `t_` で始まるか
- enum名が `e_` で始まるか
- すべての識別子がスネークケース（小文字+数字+`_`のみ）か

#### ヘッダ・ファイル
- 42ヘッダーがファイル先頭にあるか
- ヘッダのインクルードガードがあるか（`MINISHELL_H` 形式）
- .c ファイル内で構造体を宣言していないか（.h のみ許可）
- 関数内にコメントが含まれていないか

#### マクロ
- `#define` がリテラル/定数値以外に使われていないか
- マクロ名がすべて大文字か
- 複数行マクロを使っていないか

#### スペーシング
- カンマ/セミコロンの後にスペースがあるか
- 演算子の前後にスペースがあるか
- C キーワード（if, while, return 等）の後にスペースがあるか

### 🟡 重要（バグ・リーク）

#### メモリ安全性
- `malloc`/`ft_calloc` の戻り値を NULL チェックしているか
- `free` 漏れがないか（特に `ft_strjoin`/`ft_strdup`/`ft_split` の中間値）
- エラーパス（early return）で確保済みメモリを解放しているか
- `getcwd(NULL, 0)` の戻り値を free しているか

#### fd リーク
- `open` した fd を必ず `close` しているか
- `pipe` で作った fd を親子両方で適切に `close` しているか
- `dup2` 後に元の fd を `close` しているか

#### ロジック
- NULL ポインタの参照がないか（`cmd->argv[0]` が NULL のケースなど）
- 終了ステータスが正しいか（command not found=127, permission denied=126, syntax error=2）
- bash と同じ挙動をしているか

### 🔵 提案（改善案）
- 変数名・関数名が明示的か（a, b, tmp のような曖昧な名前は避ける）
- 関数が単一の責務を持っているか
- 不要なコードやデバッグ出力が残っていないか

## 3. 出力形式

問題を見つけたら以下の形式で報告してください:

```
## [ファイル名]

### 🔴 致命的（Norm 違反）
- L42: 関数 `parse_command` が 28 行（25 行以内にすること）
- L15: 変数宣言と初期化を同一行で行っている: `int i = 0;`

### 🟡 重要（バグ・リーク）
- L67: `ft_strjoin` の戻り値を free せずに次の `ft_strjoin` を呼んでいる

### 🔵 提案
- L15: 変数名 `tmp` を `expanded_value` にすると意図が明確になる
```

問題がなければ「✅ レビュー完了 — 問題なし」と報告してください。
